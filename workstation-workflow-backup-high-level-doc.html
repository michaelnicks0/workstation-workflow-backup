1 What it does

Eight capabilities across hourly automation, security hardening, and structured restore — all fail-closed and silent on success.

⏱

Hourly silent sync

systemd user timer fires at :45 every hour. rsync mirrors WSL paths; robocopy mirrors Windows artifacts over SMB. Successful runs emit nothing.

🗄

Consistent SQLite snapshots

SQLite backup API snapshots Hermes state.db, kanban.db, lifelog, and browser-memory DBs before the rsync phase — never copies a live, write-in-progress database.

🛡

Fail-closed growth guard

Checks dataset used (≤2 TiB), snapshot-held blocks (≤1 TiB), free space (≥2 TiB), and snapshot count (≤5 000) before and after writes. Stops backup; never deletes.

🔐

Restricted runtime identity

Hourly path uses dedicated user ws1backup with pinned host key (StrictHostKeyChecking=yes, HostKeyAlias) and a forced-command SSH dispatcher that rejects arbitrary shell, PTY, and rsync sender mode.

📋

Run ledger + integrity manifest

Every run appends to a SQLite event ledger. On success, SHA-256 checksums for critical restore artifacts are written to NAS under current/_manifests/integrity-manifest.json.

🔔

Failure-only Telegram alert

systemd OnFailure unit sends a Telegram message via the Hermes bot on every nonzero service exit. Zero noise on success; immediate signal on failure.

📸

ZFS snapshot ladder

TrueNAS periodic snapshot tasks retain: hourly 2 days, daily 2 weeks, weekly 8 weeks, monthly 1 year. No repo-owned zfs destroy helper.

🗺

Structured restore path

Restore runbook covers missing repo files, Hermes state.db, Windows artifacts, and staged restore with integrity-manifest checksum verification before replacing live state.

2 How it works

The same guarded path runs every hour — pre-flight, data movement, post-flight, publication.

  1. 1
    Timer fires at :45workstation-workflow-backup.timer triggers the service via systemd user session
  2. 2
    Pre-flight growth guardcheck-nas-growth-guard.sh --stage pre: fails closed if dataset used, snapshot blocks, free space, or snapshot count exceeds budget
  3. 3
    Consistent SQLite snapshotssnapshot-sqlite-dbs.py uses the SQLite backup API before any rsync — ensures consistent copies of live Hermes, lifelog, and browser-memory DBs
  4. 4
    WSL rsync to NASrsync over restricted runtime SSH (ws1backup, pinned host key, forced-command) writes ~/repos, ~/.hermes, ~/.ssh, and other paths to NAS current/wsl/
  5. 5
    Windows robocopy over SMBsync-windows-critical.ps1 via PowerShell rsyncs Desktop, Documents, .ssh, Terminal, VS Code, Chrome metadata to NAS current/windows/
  6. 6
    Post-flight growth guardcheck-nas-growth-guard.sh --stage post: re-checks all budgets after writes; fails closed if exceeded
  7. 7
    Ledger completion + manifestrecord-run-ledger.py records completed event; nas-integrity-manifest-helper.py writes SHA-256 checksums for critical restore artifacts to NAS

3 The mental model

Everything flows through one restricted path on a ZFS snapshot ladder with a fail-closed guard.

Restricted runtime identity (ws1backup). Admin SSH required only for provisioning, restore, and snapshot inspection.

6 Safety boundary

The hourly runtime path enforces a hard boundary: restricted identity, no root, pinned host key, fail-closed guards.

The runtime SSH path is forced-command only: no interactive shell, no PTY, no rsync sender mode, no port forwarding.
🛡️
Root/admin SSH is required only for explicit provisioning, snapshot inspection, and restore. See ADR-0006.
